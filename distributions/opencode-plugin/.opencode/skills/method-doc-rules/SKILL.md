# 방법 문서 작성 규칙

방법 문서를 작성할 때 따르는 세부 규칙. 기본 문체와 일반 산출물 원칙은 `general-doc-rules` skill 을 따른다.

# 개요

방법 문서는 사용자가 실제 화면과 결과를 따라가며 오류 없이 목적을 완료하도록 돕는 실행형 문서다.
독자는 해당 도구, 보드, 라이브러리를 처음 다루는 사용자라고 가정한다. 전문 용어, 약어, 내부 회로명, 프로토콜명, 페리페럴 이름은 기본적으로 뺀다. 동작 수행에 직접 필요 없는 배경 지식, 원리 설명, 기술 이유는 본문에 넣지 않는다. 용어를 넘기는 한 문장이 사용자의 다음 행동을 바꾸지 않으면 그 문장은 통째로 뺀다.

기본 문체는 상위 `general-doc-rules` 의 `문체`, `길이와 구조`, `일반화`, `코멘트와 주석` 규칙을 그대로 따른다.

# 준비

1. 문서 제목은 `[대상] X 하는 법 (환경, 도구)` 형식으로 쓴다. 대괄호 prefix 에 대상 보드/모듈/라이브러리/도메인을 넣어 검색과 분류를 돕고, 본문은 핵심 액션 (한국어) 로, 환경과 도구는 괄호 suffix 로 분리한다 (예 `[ESP32-S3] 맥에서 개발보드 셋업 하는 법 (Mac, PlatformIO)`).
2. 제목은 검색 의도와 맞는 자연어 질문형에 가깝게 작성한다.
3. 사용자별 절대 경로, 개인 ID, 토큰은 본문에 박지 않는다.
4. 시점, 버전, 플랫폼에 따라 달라지는 UI는 실제 화면이나 실행 결과를 받은 뒤 작성한다.
5. 본문의 `# 준비` 섹션은 준비물 나열이 아니라 사용자가 `# 실행` 시작 전에 수행하는 준비 액션 단계로 작성한다. 예를 들어 환경 셋업 참조 링크, 케이블 연결처럼 쓴다.

# 실행

1. 큰 틀은 반드시 `# 개요`, `# 준비`, `# 실행`, `# 확인` 순서로 작성한다.
2. 단계는 사용자가 실제로 수행하는 순서대로 작성한다.
3. 모든 단계를 한 번에 추측해서 쓰지 않는다. 첫 단계 또는 다음 단계만 작성하고, 사용자의 화면, 출력, 결과를 받은 뒤 이어 쓴다.
4. UI 식별자, 버튼, 메뉴, 체크박스, 다이얼로그 이름, 영역명은 인라인 코드로 표시한다.
5. 입력 필드 레이블은 볼드 처리 후 콜론을 붙이고, 입력값은 인라인 코드로 표시한다.
6. 한 단계 안에 두 개 이상의 순차 액션이 있으면 들여쓴 번호 매기기 list 로 분해한다.
7. UI 입력 폼처럼 동시에 입력하는 항목은 일반 글머리 기호로 표시하고, 부모 액션 아래에 중첩한다.
8. 메인 절차에서 벗어나는 오류 대응, 환경별 예외, 우회 방법은 노란색 ⚠️ 콜아웃으로 분리한다. 콜아웃 자체가 참고사항임을 보여주므로 제목에 `참고` 라는 단어를 붙이지 않는다.
9. 괄호 안 부연은 간결하게 쓴다. 예를 들어 `처음이면 몇 분 걸릴 수 있다` 대신 `초기 설정의 경우 수 분 소요`처럼 쓴다.
10. 입력 항목 불릿 안에서는 문장형 종결을 생략하거나 명사형으로 줄인다.
11. 실행 결과 (터미널 출력, 시리얼 모니터 출력, 결과 화면 스냅) 는 `**✅ Output**` 라벨 아래에 `bash` 언어로 지정된 코드블록을 두고, 그 코드블록을 quote (인용) 블록으로 감싼다. 언어를 지정하지 않으면 Notion 이 임의 언어 (javascript 등) 로 감지해 하이라이트가 틀어진다. 라벨과 quote 블록 모두 부모 번호 항목보다 한 단계 들여쓴다. 의도된 렌더링 결과는 출력 코드블록 좌측에 quote 의 세로선이 표시되어 입력 명령 코드블록과 시각적으로 구분되는 형태다. quote 의 세로선이 보이지 않으면 인용 처리가 빠진 것이므로 출력 코드블록을 quote 블록 안에 다시 넣는다. 결과 내용은 반드시 실제 실행으로 받은 값으로 채우고, 소스 코드/문서를 보고 추측해서 만들지 않는다. 실제 결과가 없는 상태에서는 `**✅ Output**` 라벨과 quote 블록을 아예 생략하거나 다음 단계로 미루고, 출력을 받은 뒤 채운다.

# 목록과 코드블록

1. 번호 목록이나 글머리 기호 아래에 코드블록, 콜아웃, 추가 설명을 넣을 때는 반드시 부모 항목보다 한 단계 더 들여쓴다.
2. 들여쓰기하지 않은 코드블록은 목록을 끊어 다음 번호가 `1` 로 다시 시작하게 만들 수 있으므로 금지한다.
3. 코드블록을 목록 항목의 일부로 유지할 때는 `1. 설명` 다음 줄을 비우고, 코드블록 전체를 스페이스 4칸 또는 탭 1개로 들여쓴다.
4. 코드블록을 들여쓰기하기 어렵거나 Notion 렌더링이 불안정하면 번호 목록 안에 넣지 말고, 목록을 짧게 끝낸 뒤 별도 문단의 `명령` 또는 `예시` 블록으로 분리한다.
5. 같은 절차 안에서 코드블록 때문에 번호가 끊겼는지 fetch 결과 또는 화면에서 반드시 확인한다.
6. 특히 명령어 예시가 많은 문서는 최종 확인 때 번호가 `1, 1, 2, 1` 처럼 재시작하지 않는지 확인한다.
7. 한 섹션 안에서 대부분의 단계가 `명령`, `예상 출력`, `파일`, `예시`, `콜아웃` 을 포함해도 자동 번호 목록을 유지한다. 번호 목록을 없애지 말고 모든 하위 블록을 해당 번호 항목 아래에 처음부터 들여쓴다.
8. 번호 목록 항목 아래의 사용자가 터미널/세션에 입력하는 명령 코드블록은 별도 라벨 (`**명령**` 등) 없이 단계 설명 바로 아래에 둔다. `파일`, `예시` 같이 명령이 아닌 코드는 부모 번호 항목보다 한 단계 들여쓴 굵은 라벨로 둔다.
9. 번호 목록 항목 아래의 코드블록은 부모 번호 항목보다 한 단계 들여쓴다. 라벨이 붙는 경우 (`파일`, `예시`) 는 라벨 아래에 둔다.
10. 번호 목록 항목 아래의 콜아웃은 `<callout>` 시작 태그부터 `</callout>` 종료 태그까지 전체를 부모 번호 항목보다 한 단계 들여쓴다. 콜아웃 내부 본문은 콜아웃 태그보다 한 단계 더 들여쓴다.
11. 링크만 단독으로 `링크` 라고 쓰지 않는다. 액션 문장 안에 `[링크]` 를 붙이거나 `설치 페이지 [링크]` 처럼 한 줄로 쓴다.
12. 같은 명령을 명령 코드블록과 `**✅ Output**` 코드블록 사이에 중복해서 쓰지 않는다. 명령 코드블록은 사용자가 입력할 명령만, `**✅ Output**` 코드블록은 출력 예시만 담는다.
13. 코드블록 바로 다음에 콜아웃을 붙여 화면에서 겹치거나 붙어 보이면, 콜아웃 앞에 들여쓴 일반 문장을 하나 둔다. 예 `아래 오류가 나오면 Docker Desktop 실행 상태 확인.`.
14. Notion 화면에서 번호가 모두 `1` 로 보이거나 코드블록과 콜아웃이 겹치면 문서 형식 실패로 간주한다. 이때 자동 번호 목록을 제거하지 말고, 하위 블록의 들여쓰기를 고쳐 번호 목록 안에 다시 포함시킨다.

# 표현

1. 본문은 명사/체언 종결을 우선한다 (예 `~연결`, `~사용`, `~확인`, `~측정`). 동사형 종결 (`~한다.`) 은 핵심 단일 액션 한 줄에만 둔다.
2. `# 개요` 는 `~하는 테스트`, `~을 확인하는 것` 같은 메타 종결 대신 `~확인`, `~측정` 같은 명사 종결으로 끝낸다.
3. 조사 (을/를/의/이/가) 는 가독성을 해치지 않는 선에서 생략한다 (예 `Mac 을 연결` → `Mac 연결`).
4. 부연 설명은 별도 문장이 아니라 본문 끝 괄호로 흡수한다 (예 `풀-다운 저항 회로를 만든다. (신호 전압으로 GPIO5 핀 사용)`).
5. 본문 링크 표시 텍스트는 페이지 제목을 박지 않고 `[링크]`, `[참조]` 같이 짧게 둔다. 어떤 페이지인지는 클릭으로 확인하게 한다.
6. 사진, 회로도가 보여주는 디테일 (결선 위치, 프로브 색깔, 부품 위치, 부품 수량) 은 본문 단계로 분해하지 않는다. 본문은 한 줄 액션만 두고 시각 자료에 맡긴다.
7. 다른 페이지 셋업이 제공하는 정보 (공통 라이브러리, 공통 설정, 프로젝트 구조) 는 참조 링크로 승계하고, 세부 의존 내용을 본문에 다시 적지 않는다.

# 확인

1. 문서 마지막에는 반드시 확인 섹션을 둔다.
2. 확인 섹션은 실제 완료 여부를 판단할 수 있는 출력, 화면 상태, 성공 조건을 다룬다.
3. 성공 기준은 큰 흐름 단위 끝에만 둔다. 개별 액션마다 붙이지 않는다.
4. 성공 기준은 초록색 ✅ 콜아웃으로 표시한다. 초록색 콜아웃과 ✅ 아이콘 자체가 성공 기준임을 보여주므로 `성공 기준` 같은 라벨은 본문에 붙이지 않고 바로 상태 설명을 적는다.

```
<callout icon="✅" color="green_bg">
    사용자가 문서의 목적을 완료했음을 판단할 수 있는 상태를 적는다.
</callout>
```

# 개요

1. 의존 셋업 [링크].
2. 케이블/보드 실물 연결.

1. 첫 액션.
2. 다음 액션.

# 확인

<callout icon="✅" color="green_bg">
    사용자가 목적을 완료했음을 판단할 수 있는 상태.
</callout>
````

## 명령 + ✅ Output 쌍

번호 항목 본문 아래에 명령 코드블록, `**✅ Output**` 라벨, 출력 코드블록을 모두 한 단계 들여쓴다. 출력 코드블록은 quote (인용) 블록으로 감싸 좌측에 세로선이 보이게 한다. 명령 코드블록에는 라벨을 붙이지 않는다.

````markdown
1. Docker CLI 서버 연결 확인.

    ```bash
    docker --version
    docker-compose version
    colima version
    ```

    **✅ Output**

    > ```bash
    > Docker version 29.4.3, build 055a478ea9
    > Docker Compose version 5.1.3
    > colima version 0.10.1
    > ```
````

## ⚠️ 오류 콜아웃

메인 절차에서 벗어나는 예외, 오류 대응, 환경별 우회. 제목에 `참고` 같은 단어를 붙이지 않는다. 번호 항목 안에 둘 경우 콜아웃 태그 전체를 한 단계 들여쓴다.

````markdown
<callout icon="⚠️" color="yellow_bg">
    `docker: Cannot connect to the Docker daemon` 오류 시 Colima 데몬 재시작 필요.
</callout>
````
